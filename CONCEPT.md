## 一、Auftrag 是什么，从哪来

**Auftragstaktik（英文叫 mission command，任务式指挥）** 是普鲁士军队在 19 世纪发展出来的一套指挥方式。

它的来历是被打出来的。1806 年普鲁士在耶拿被拿破仑打得惨败，之后 Scharnhorst、Gneisenau 一路到老毛奇（Moltke the Elder）反思出一个结论，老毛奇那句话是整套东西的地基：

> **没有任何计划能在与敌接触后存活。**

意思是：战场情况在你下命令的那一刻就已经开始变了。等前线把新情况上报、中央重新决策、命令再逐级传回去——等命令到了，前线的真实情况又变了。**中央的决策带宽，天生就是整个系统的瓶颈。**

唯一的解法：**把决策权下放到信息最新鲜的地方**，也就是前线那个人手里。别一步步遥控他，而是把「你到底要达成什么、为什么」告诉他，给他行动自由。情况一变，他基于这个共同目标自己修正，而不是每次偏离都跑回来请示。

### 一个最容易被误解的点

Auftrag **不是「少控制」，是把控制从「过程」搬到「意图 + 边界」上。**

它不是放手不管。它是换了个地方管：不管你具体怎么走每一步（过程），只死死咬住「要到哪」（意图）和「哪几条绝不能碰」（边界）。中间怎么走，交给执行的人自己判断。

所以它省的不是控制，**省的是带宽**——你用一次把意图说透的前期成本，换掉了无数次「偏离 → 请示 → 等待 → 再下达」的持续消耗。

### 它有个前置条件，而且这个条件最容易被漏掉

Auftrag 能跑，靠的**不是自由本身**，而是一套共享的前提：大家受过同一套训练、脑子里装着同一个目标、彼此信任。正因为共享，十个下属各自独立做的判断才会**收敛到同一个终点**，而不是各飞各的。

这个条件一旦不满足，Auftrag 立刻变成灾难:给一个判断力不够、又不和你共享前提的下属充分自由，你得到的不是灵活应变，而是**自信地做错**。

### 搬到 agent 上，为什么这套特别值得做

给 agent 派活，本质就是在做 mission command:给它清晰的目标、原因、边界、升级条件,而不是写一大堆过程指令。但 agent 和人类下属有两个关键区别,恰恰让「意图」「边界」「升级条件」比管人时**更重要**:

1. **agent 会自信地失败。** 人类下属不确定的时候通常会说一声,所以「他没吭声 = 大概没事」。agent 反过来——它沉默不代表成功,可能是它自信地做错了还不知道。所以「什么情况该停下来问」必须**显式写出来**,不能指望它自觉。

2. **agent 遇到模糊地带,会朝「它训练数据里的平均答案」填空,而不是朝「你的语境」填空。** 人类下属因为和你共享前提,猜测会偏向你;agent 的猜测偏向通用均值——听着合理,但对你是错的。所以**边界必须写明**,不能靠它「懂你」。

这两点就是这个 skill 存在的全部理由:**在派活的那一刻,把意图、边界、升级条件说透,让 agent 的自主执行收敛而不是发散。**

## 二、一份 Auftrag 的五个格子

一份完整的 Auftrag 有五格,缺一不可。它是一份**双向契约**:你负责把它写清楚,agent 负责照着它执行,两边都咬住这五格。

### 1. 意图 / 为什么（Intent）—— 最重的一格

不只说「这个任务是什么」,还要说清**这任务服务于什么、再上一层又想达成什么**。

为什么它最重:当 agent 手里的「具体怎么做」失效了(计划撞上现实),**唯一能让它自己重新推导出正确动作的,就是这个「为什么」**。没有「为什么」,过程指令一断,它就两手空空——要么卡住,要么乱来。意图是那根让独立判断仍然收敛到同一个终点的线。

这一格也是**你最强、最不该外包**的一格。它必须从你脑子里出来。

### 2. 目标状态 / 完成长什么样（Target state）

描述的是**要达成的状态,不是要执行的步骤**。

是「夺取那座桥」,不是「沿这条路、按这个队形、这个时间走」。你说状态,把怎么到那个状态的自由留给 agent。

### 3. 边界 / 红线（Boundaries）

**只写那几条绝不可逾越的硬约束,其余全是自由。**

关键在**最小化**。每多写一条过程指令,你就多还原了一分「中央遥控」的脆弱性——你又变回那个一步步下命令的人了。边界只写「整体目标确实要求」的那几条,别把偏好、习惯、锦上添花的东西塞进来当红线。

### 4. 升级条件 / 什么情况停下来找我（Escalation）

**明确写出:在什么情况下,agent 不该自己往下做,而要停下、把决策交还给你。**

这是授权的边界线,也是 agent 派活里**最常被漏掉**的一格——大家拼命写 agent 该做什么,却很少写「什么情况下它该停下来问」。而对 agent 来说这一格尤其要紧,因为它不会像人一样自觉地喊停(见第一部分)。

对不同类型的任务,这一格内容差别很大。比如调研类任务,一条典型的升级条件是「遇到不认识的组件/工具,停下来列出来问我,绝不自己猜」。

### 5. 完成标准 / 怎么算做完、要什么证据（Done-criteria）

**证明「达到了目标状态」的具体核验是什么。** 不是「我觉得做完了」,而是「跑了什么检查、得到什么结果」。

这一格是你**最想跳过、又最不能丢**的。跳过它,是因为写它逼你在动手之前就把终点想清楚,很烦。但它不能丢,因为:

- 「做不对会自己吭声」那套机制,全靠有个「完成长什么样」当对照物。没有它,agent 不知道该在哪一刻喊停——它没有参照系。
- 它是这份契约里最硬、最可审计的一格。

## 三、这个 Skill 的构想（三步结构）

这个 skill 要做的事很朴素:**在你派活之前,逼你(在它的引导下)把上面五格填扎实,然后沉淀成清单,最后问你要不要现在就派。**

它不是一套多 agent 系统,不是常驻的助手/管家角色。它就是一个**你每次要认真派个活时会调用的引导流程**。

### 第一步:引导你填出一份扎实的 Auftrag

**核心不是一张表格,是一次能追问的访谈。**

表格只能查你「填没填」,查不出你「填得敷衍不敷衍」。而返工几乎全来自后者——不是你漏了「为什么」,是你「为什么」那栏写了句正确的废话就过去了。所以这个 skill 真正比「你自己对着模板写」强的地方,就一个:**你答得浅,它追一句。**

这一步的重心因此不在那五个格子,在那几句「你想糊弄过去时它不放你走」的追问。尤其是两格最容易注水:

- **为什么**(意图)
- **什么情况停下来找我**(升级条件)

这两格答实了,别的基本自然就好。判断「能不能往下走」,不看格子空不空,看这两格实不实。**完成标准那一格空着,不许往下走。**

### 第二步:沉淀成任务清单(并顺便当筛子)

填完的 Auftrag 落进 Todoist(把 brief 挂在任务描述里)。这步机械。

但它有个附加作用:**认真填「为什么」的过程,会当场帮你分流。** 你写着写着可能会发现——

- 「这事的价值其实是让我自己学会,不是产出个东西」→ 那它不该派 agent,得留你自己串行地做。
- 「这根本是个决定,不是个任务」→ 那它是你自己的活,不是委托对象。

所以这个 skill 不只帮你写 brief,它还在你**浪费一个 agent 之前**,先把那些根本不该委托的挑出来。你不用专门去做任务分类,老实填 Auftrag 的过程里它自己就分好了。

### 第三步:可选是否开始

**写 brief 和派活是两个动作,中间断开。**

一份 Auftrag 填完 → 问你一句:「现在就派个 agent 去干,还是先存着?」

- 存着 → 进清单,等以后。
- 现在干 → 把这份 Auftrag 甩给一个 agent 当执行指令。

## 四、v0 刻意不做的事(给 Claude Code 的边界)

这几条是硬边界,是这份文档作为 Auftrag 给你(Claude Code)划的红线。设计时请咬住:

1. **不替用户起草 Auftrag。** skill 只负责**问**,用户负责**答**。一旦让它写初稿,用户会顺手点头通过一个平庸的版本,而「意图」是用户最强、最不该外包的部分。**只问,不写。**

2. **别急着上 hook、上校验脚本、上自动化强制。** 这些等真用过十几回、发现哪里老漏,再精准补。v0 越小越好。

3. **别把它做成一套「系统」。** 它不该引入需要维护的常驻 agent、agent 之间的通信、状态同步。它就是一个引导流程 + 一份模板 + 落进 Todoist + 结尾问一句。

4. **别把边界那一格做成一个大清单。** 这个 skill 自己也要吃自己的狗粮——它引导用户写的「边界」要最小化,它自身的复杂度也要最小化。

## 五、留给 Claude Code 探索的空间

以下这些**故意没写死**,请你在理解上面意图的前提下自己设计、并在动手前把你的设计思路讲给用户确认:

- 那几句关键追问具体怎么措辞?怎么判断用户「答浅了」需要追问?
- 访谈的交互形态——是一格一格问,还是先让用户一口气写、再逐格追问?
- 模板/brief 用什么结构落盘、落在哪。
- 和 Todoist 的对接方式。
- 「完成标准空着不许往下走」这类硬规则,用什么方式实现比较自然而不烦人。
- 这个 skill 怎么和用户已有的 CLAUDE.md、既有 skills(那套共享前提)衔接——brief 里的边界和完成标准应该**引用**它们,而不是重复。

一句话交底:**你要建的是「一次能把活派对」的最小引导,不是一套 agent 架构。** 判断你设计得对不对,标准就一个——它有没有让用户在派活前,把「为什么」和「什么情况停下来找我」这两格真正想清楚。
